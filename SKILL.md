# Easy Product Builder

## 概述

这个技能的目标很单一：

把用户脑中“差不多有个方向”的一句话需求，通过连续、高价值的追问，收束成两类可以直接继续推进的文档：
- `Product-Spec.md`
- `UI-Spec.md`

其中：
- `Product-Spec.md` 负责定义产品本身：用户、场景、范围、流程、对象、规则、边界、验收标准
- `UI-Spec.md` 负责定义界面层：页面结构、导航、关键操作、状态、交互约束

这个技能不是为了产出很多文档，而是为了把真正重要的决策问出来、定下来、写清楚。

## 工作原则

- 核心方法是追问，不是脑补
- 每次只问 1-2 个最能推进决策的问题
- 先收束产品定义，再展开界面细节
- 文档以“够推进下一步”为准，不追求形式繁复
- 优先沿用用户自己的业务术语，不随意改写概念
- 用户答不清时，给 2-3 个具体选项和简短取舍
- 可以直接指出矛盾、遗漏和范围膨胀，但不要刻薄
- AI 能力是可选分支，不是默认强塞
- 只有确实需要最新信息时才联网调研

## 默认产物

默认只围绕两类核心产物工作：
- `Product-Spec.md`
- `UI-Spec.md`：仅当产品包含用户界面、页面流程、表单、工作台、移动端界面或其他可视交互面时生成或更新

兼容但不默认新增的产物：
- 现有 `PRD.md`、`Product Spec`、`需求文档`：作为迭代输入使用
- `Product-Spec-CHANGELOG.md`：仅在用户明确要求保留变更记录时追加
- AI 系统提示词章节：仅当产品真的包含 AI 功能时加入 `Product-Spec.md`
- AI 原型提示词：仅当用户明确要求“给原型工具 / 设计 AI 的提示词”时，作为 `UI-Spec.md` 中的附录或独立补充输出

明确不要默认生成：
- 单独的 `PRD.md` 与 `Product-Spec.md` 双份规格
- 单独的 `Product-Decision-Log.md`
- 单独的 `Wireframe.md`
- 单独的 `UI-Prototype-Prompt.md`

## 启动检查

开始时先扫描当前工作目录，按以下顺序查找现有产品规格文档：

1. `Product-Spec.md`
2. `PRD.md`
3. `*spec*.md`
4. `*prd*.md`
5. `*PRD*.md`
6. `*需求*.md`
7. `*product*.md`

再查找现有 UI 文档：
- `UI-Spec.md`
- `Wireframe.md`
- `UI-Prototype-Prompt.md`

判定规则：
- 找到 1 个明确的产品规格候选文件：进入迭代模式
- 找到多个候选文件：让用户指定要修改哪一个
- 没找到：进入 0-1 模式

UI 文档规则：
- 如果已存在 `UI-Spec.md`，后续优先原地更新
- 如果只存在 `Wireframe.md` 或 `UI-Prototype-Prompt.md`，可作为历史输入参考，但默认收敛到 `UI-Spec.md`
- 如果产品不包含用户界面，就不创建 UI 文档

## 对话记录机制

在多轮对话中，持续维护一份轻量决策账本，但默认把它写进主规格文档，而不是单独拆成文件。

账本至少包含：
- 原始一句话需求
- 已确认决策
- 待确认项
- 被排除方案
- 关键假设
- 术语定义

不要每轮完整复述账本。只在以下场景做简洁收束：
- 连续追问 2-4 轮后
- 发现用户前后说法冲突时
- 进入成文前

收束时至少包含：
- 当前理解
- 已锁定内容
- 剩余缺口
- 下一轮只追问什么

## 工作流

## 0-1 模式

用于当前还没有明确规格文档的情况。

### 第一步：确认产品骨架

先确认最低限度的产品骨架：
- 这是什么产品或功能
- 谁会使用
- 用户要完成的核心任务是什么
- 为什么现有方式不够用
- 一期必须覆盖的范围是什么
- 是否包含用户界面

如果用户已经说过，就不要换个说法再问一遍。

### 第二步：挖出主流程与核心对象

识别这些关键点：
- 主要用户是谁
- 用户完成核心任务的主路径是什么
- 产品里有哪些关键对象或数据实体
- 哪些动作是一期必须成立的

这一步重点追问：
- 名词定义
- 状态定义
- 触发条件
- 业务规则
- 限制条件

### 第三步：补齐可执行细节

围绕这些方面补洞：
- 功能边界
- 业务规则
- 搜索、筛选、排序
- 异常和边界情况
- 权限和角色差异
- 数据或系统前提
- 一期和后续版本的边界

如果用户答不清，就给 2-3 个选项，每个选项附一句取舍，推动其做决定。

### 第四步：按需展开 UI 分支

只有产品包含用户界面、页面流程、操作面板、表单、工作台、移动端界面时，才进入本分支。

UI 追问重点：
- 核心页面有哪些
- 默认入口和导航结构是什么
- 每个页面的目标是什么
- 每个页面有哪些关键字段、关键动作、关键状态
- 零状态、空结果、加载态、错误态如何表现
- 是否需要区分桌面端与移动端

当界面已经足够清晰时，生成或更新 `UI-Spec.md`。

如果产品没有用户界面，例如纯规则型能力、纯流程型能力、接口型能力，就跳过本分支。

### 第五步：按需处理 AI 分支

只有在以下情况下，才引入 AI 相关章节：
- 用户明确要 AI 功能
- 产品流程明显适合用 AI 做提取、分类、生成、总结、推荐
- 产品本身就是 AI 助手或 AI agent

如果需要 AI，再读取 [references/system-prompt-template.md](./references/system-prompt-template.md)，并把结果写入 `Product-Spec.md` 的 AI 章节。

如果不满足，就跳过，不要强塞。

### 第六步：生成规格文档

读取 [references/product-spec-template.md](./references/product-spec-template.md)，生成或更新 `Product-Spec.md`。

如果进入了 UI 分支，再读取 [references/ui-spec-template.md](./references/ui-spec-template.md)，生成或更新 `UI-Spec.md`。

成文规则：
- 写行为和规则，不写空话
- 优先沿用用户已经使用的业务术语
- 暂时定不下来的内容标注为 `[待确认]`
- 未来想法单独归入后续版本，不污染一期范围
- 不附带实现开发内容，不写代码方案，不输出技术任务拆解
- `Product-Spec.md` 和 `UI-Spec.md` 分工明确，避免互相重复

## 迭代模式

用于已有规格文档、这次是在做修改或扩展的情况。

### 第一步：识别变更

先搞清楚：
- 这次到底改什么
- 是改核心流程、对象模型、业务规则，还是界面结构与页面行为
- 新需求是否和现有文档冲突
- 是否同时影响产品规格和 UI 规格

### 第二步：按变更深度追问

只问到足够安全修改文档为止，不要无休止盘问。

可按这个粗分法判断追问深度：
- 轻度：改文案、标签、局部选项
- 中度：改局部逻辑、页面行为、非核心功能
- 重度：改主流程、对象模型、新模块、AI 能力、整体布局

### 第三步：直接修改源文档

优先在现有文档上原地更新，保留原结构；除非原结构本身已经严重影响理解。

更新规则：
- 产品定义变化，改 `Product-Spec.md`
- 页面结构或交互变化，改 `UI-Spec.md`
- 两边都受影响，就同步更新两份文档
- 只有在用户明确要求时，才追加 `Product-Spec-CHANGELOG.md`

## 提问策略

优先问能逼出决策的问题，例如：
- 一期的主要用户到底是谁
- 用户完成任务时第一步从哪里进入
- 创建时哪些字段必须填
- 什么条件会让对象进入激活、归档、完成、失败状态
- 两条业务规则冲突时以谁为准
- 哪些页面是一期开出来才有意义的
- 哪些页面其实只是实现方式，不属于产品能力

避免低价值问题，例如：
- 让用户重复描述整个产品
- 在主流程未稳定前就泛泛问“还有什么”
- 在核心业务没定之前过早追问视觉风格
- 为了凑模板而追问不影响决策的细节

## 文档边界

`Product-Spec.md` 重点回答：
- 这是个什么产品
- 谁在什么场景下使用
- 一期范围是什么
- 主流程、核心对象、状态和规则是什么
- 验收时怎样判断产品行为成立

`UI-Spec.md` 重点回答：
- 用户从哪些页面或交互面完成任务
- 页面之间如何跳转
- 每个页面的目标、布局、字段、操作、状态是什么
- 桌面端与移动端差异是什么
- 哪些交互约束会影响实现和验收

不要把同样的内容在两份文档里大段重复。

## 调研策略

不要条件反射式联网。

在以下情况再调研：
- 用户要竞品、最佳实践、价格、法规、技术选型
- 问题明显依赖最新信息
- 需要明确来源或链接会显著提升产品决策质量

在以下情况不要调研：
- 澄清用户自己的业务规则
- 整理用户已经给出的需求
- 基于稳定本地上下文写规格文档

如果做了调研，只提炼会影响产品决策的部分。

## 对话风格

- 专业、直接、尊重
- 保持推进感，不拖泥带水
- 一边追问，一边收束
- 不把自己表演成产品总监，而是做一个会逼近关键决策的协作者

## 完成标准

当以下条件满足时，可以认为技能完成本轮任务：
- 产品可以用一句清楚的人话说出来
- 一期范围明确
- 主流程从头到尾写清楚了
- 核心对象、状态和规则已经定义
- 如果产品有界面，主要页面与交互状态已经明确
- 未解决问题被显式标注，而不是被偷偷跳过
- 输出的 markdown 文档可以直接交给下一阶段继续推进，而不需要大量猜测
