# easy-product-builder

`easy-product-builder` 是一个用于需求澄清与产品规格生成的 skill。

它面向这样一类问题：用户脑子里已经有产品方向，但表达仍然零散、边界不清、流程不稳，直接进入设计或开发很容易走偏。这个 skill 的作用，就是通过结构化追问，把这些模糊想法收束成一份清晰、可执行的 `Product-Spec.md`。

## 亮点

- 以追问驱动需求收束，而不是直接替用户脑补答案
- 强调结构化决策，帮助用户把模糊想法变成明确规则
- 对话风格专业、直接、尊重，不依赖刻薄或攻击式表达
- AI 功能按需启用，不会默认把所有产品都写成 AI 项目
- 调研策略克制，只有在确实依赖最新信息时才联网
- 产出面向执行，适合继续衔接设计、开发、任务拆解和后续迭代

## 适用场景

- 从 0 到 1 设计一个新产品、应用、工具或内部系统
- 对已有产品新增功能并更新规格文档
- 将聊天式需求讨论整理成结构化文档
- 在进入设计或开发前，先把核心流程、页面和规则定清楚

## 核心功能

### 1. 需求澄清

通过连续但克制的高价值追问，确认：
- 产品定位
- 目标用户
- 核心问题
- 主要使用场景
- 一期范围

### 2. 核心流程抽取

将用户零散的描述整理为：
- 主路径
- 关键对象
- 状态定义
- 触发条件
- 业务规则

### 3. 关键缺口识别

主动发现规格文档最容易漏掉的部分，例如：
- 数据模型
- 搜索、筛选、排序
- 归档、删除、恢复规则
- 页面结构与交互面
- 边界情况与异常处理

### 4. 选项式决策推进

当用户说不清楚时，不停留在“你再想想”，而是给出 2-3 个具体选项和简短取舍，推动需求继续落地。

### 5. 规格文档生成与迭代

支持两种工作模式：
- 0-1 模式：生成新的 `Product-Spec.md`
- 迭代模式：直接更新已有规格文档，并在必要时追加变更记录

### 6. AI 功能按需扩展

如果产品真的包含 AI 能力，这个 skill 可以继续补充：
- AI 功能章节
- AI 系统提示词模板
- AI 输入输出约束

如果产品不需要 AI，就不会强行加入这些内容。

## 工作方式

### 0-1 模式

当目录下没有现成规格文档时，skill 会：

1. 确认产品框架
2. 抽取核心流程
3. 补齐可执行细节
4. 按需处理 AI 分支
5. 生成完整规格文档

### 迭代模式

当目录下已经存在规格文档时，skill 会：

1. 识别变更内容
2. 按变更深度追问
3. 在源文档上直接修改
4. 必要时追加变更记录

## 文件说明

- `SKILL.md`
  技能主体定义，包含工作原则、工作流、提问策略、调研策略和完成标准。

- `references/product-spec-template.md`
  默认 Product Spec 模板，用于生成新规格文档。

- `references/changelog-template.md`
  规格文档在迭代过程中的变更记录模板。

- `references/system-prompt-template.md`
  产品包含 AI 功能时使用的系统提示词模板。

- `agents/openai.yaml`
  用于描述 skill 的展示元数据。

## 使用方法

### 在 Codex 中使用

将 skill 放在可被 Codex 发现的 skills 目录中，然后直接用自然语言触发与产品规格相关的任务，例如：

- “帮我把这个产品想法整理成 Product Spec”
- “我有一个内部工具需求，帮我梳理一期范围”
- “帮我更新现有 PRD，增加补货逻辑”

### 推荐交互方式

为了让输出质量更高，建议用户尽量先提供这些信息中的一部分：
- 产品大致是什么
- 谁会使用
- 解决什么问题
- 已经确定的功能点
- 当前最不清楚的部分

即使信息不完整，skill 也会通过追问继续补齐。

## 产出内容

根据任务不同，可能输出：
- `Product-Spec.md`
- 规格文档更新稿
- `Product-Spec-CHANGELOG.md`
- 页面结构说明
- 开发任务拆解
- 线框说明

## 适合的团队阶段

这个 skill 特别适合这些阶段使用：
- 想法刚成型，准备进入产品定义
- 需求讨论很多，但结论还没沉淀成文档
- 已有规格，但准备做中大规模迭代
- 希望在开发前把规则和边界先定清楚
